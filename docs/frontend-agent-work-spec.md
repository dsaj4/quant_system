## 工作定位

这次不是简单给原页面换皮，也不是继续用卡片堆模块。核心任务是调用前端设计相关 skill，重新规划量化策略管理台的页面切换、布局结构、模块组织、跳转形式和整体网页风格，把已经打通的后端能力组织成更适合现场演示的后台工作台。

```text
已有后端能力
  -> 管理台信息架构重排
  -> 模块级 UI/交互重新设计
  -> 更顺的后台演示操作路径
  -> 更稳的客户报告展示
  -> 浏览器验证和交付说明
```

原有页面可以被直接替代，不要求保留当前单页卡片式组织方式。但凡涉及后端 API、数据字段、数据库结构、业务规则或数据契约变化，必须先写清楚修改草案，并获得人工明确批准。

## 必须遵守的边界

- 不接真实券商，不做自动下单。
- 不把 paper run 描述成实盘交易。
- 不新增付费数据源调用，不把 JQData/BaoStock 标记为已可用。
- 不在前端、文档或日志里写入真实 token、密钥、私有分享链接。
- 不切换默认数据库，不做破坏性 migration。
- 不修改后端业务规则、API 契约或数据字段，除非先写修改草案并获得人工批准。
- 老 snapshot 必须继续可打开，展示端要对缺失字段有兜底。

## 推荐使用的技能

- `frontend-skill-router`：必须先调用，用于前端 UI/UX 任务分流。
- `ui-ux-pro-max`：用于信息架构、页面切换、布局、组件规范、交互状态和设计系统规划。
- `frontend-design`：用于正式实现新的页面、组件、仪表盘和视觉风格。
- `browser:control-in-app-browser`：用于本地页面桌面/移动验证和截图。
- `beginner-code-logic-reader`：用于解释现有页面和 API 如何串联。
- `grill-me`：用于在正式改动前对齐细节和验收标准。
- `search-layer` 或网页搜索能力：用于调研网络上类似量化、投研、交易、数据工作台模块的 UI 设计。

## 工作流

每一轮正式改动都走：

```text
系统分析
  -> 同类产品/模块 UI 调研
  -> 信息架构和视觉方向草案
  -> 修改草案
  -> 人工批准
  -> 正式修改
  -> 验证
  -> 记录结果
```

若用户直接要求“正式修改”，也要先用一小段详细规范对齐本轮范围，再实施。不要把前端重构顺手扩成后端重构。

## 重构级别

本轮按“大幅度前端重构”处理：

- 可以替换 `frontend-admin/src/App.tsx` 里的原有页面组织方式。
- 可以把原来的模块卡片重组为更清晰的导航、工作区、详情页、抽屉、步骤流、表格页或分屏布局。
- 可以调整页面视觉风格、布局密度、组件组合和交互路径。
- 可以新增前端内部组件、样式文件、工具函数和前端状态组织。
- 可以重排管理台的信息架构，让演示路径更像专业后台产品。
- 不允许未批准地改变后端 API、数据表、回测规则、数据源语义或报告 payload 契约。

## 调研要求

正式设计每个模块前，必须做轻量但真实的网络调研。调研不是复制竞品，而是提炼适合本项目的后台交互模式。

每个核心模块至少调研 2 到 3 个类似功能的产品或页面模式，并记录：

- 模块名称：例如行情数据、策略参数、回测任务、报告发布、模拟运行、操作日志。
- 参考对象：产品、开源系统、SaaS 后台、券商/量化平台截图或文档。
- 可借鉴点：导航结构、筛选方式、表格密度、任务状态、图表组织、详情展开、错误反馈。
- 不采用点：不适合本项目演示、过度复杂、偏实盘交易、合规风险高或与后端契约冲突的设计。
- 本项目落地方案：在统一风格下如何改该模块的 UI 和交互。

调研输出可以写在修改草案中；如果内容较多，可新增 `docs/frontend-rebuild-research.md`。但最终项目文档仍需保持精简，避免堆过期资料。

## 优先级

### P0：量化策略管理台整体重构

目标：让后台演示路径从“单页卡片堆功能”升级为“专业量化工作台”。

重点：

- 重新规划管理台页面结构，不再简单用卡片承载所有模块。
- 设计清晰的导航方式：侧边栏、顶部导航、工作流步骤、详情页或标签页可按调研结果选择。
- 把高频演示路径放在明显位置：数据、策略、回测、报告、paper run。
- 把低频管理能力收纳到二级区域：操作日志、迁移/健康、provider 预留说明。
- 每个模块要有统一但不僵硬的布局规则：列表、详情、状态、操作区、反馈区。
- 关键操作要有演示友好的反馈：创建成功、任务运行中、失败原因、复制链接、发布完成。
- 原页面可以被替换，不要求保持当前卡片形态。

### P1：管理台模块级 UI/交互设计

目标：针对每个模块先调研类似功能，再在统一风格下逐个设计。

推荐模块拆分：

- 仪表盘总览：系统状态、数据源状态、最近回测、最近报告、paper run 概览。
- 行情数据：标的选择、provider、frequency、adjust、导入任务、数据质量。
- 策略参数：rolling_t_grid 参数集、默认值、风险提示、参数复用。
- 回测任务：创建回测、任务列表、结果摘要、曲线和交易入口。
- 报告发布：快照列表、发布状态、分享链接、撤销和复制。
- 模拟运行：paper run 状态流转、信号、模拟成交、失败原因。
- 系统审计：操作日志、health、schema/migration 状态。

每个模块交付前都要说明：

- 参考了哪些同类 UI 模式。
- 本模块采用什么布局和交互。
- 如何服务 demo 展示。
- 是否需要后端契约变化；如果需要，必须等待批准。

### P2：客户展示端

目标：让客户第一眼看到的是“成熟投研报告”，不是“工程调试页面”。

重点：

- 报告首屏突出策略、标的、区间、数据源、复权、核心结论。
- 负收益和回撤要有专业解释，不回避结果。
- 指标区分主要指标和辅助指标，避免所有数字同权重。
- K 线、权益曲线、回撤、仓位、信号摘要布局清楚。
- 交易明细默认不要撑长整页，可摘要、滚动、折叠或只突出关键交易。
- 风险披露和数据质量要容易看到，但不喧宾夺主。
- token 错误、加载中、无数据、旧快照字段缺失都要有稳定页面状态。

### P3：后台演示工作流

目标：让演示人员顺着一个自然路径完成操作。

推荐路径：

```text
选择/创建标的
  -> 拉取行情
  -> 查看数据任务和质量
  -> 选择策略参数
  -> 跑回测
  -> 发布快照
  -> 复制分享链接
  -> 查看 paper run 状态
```

重点：

- Tushare 作为默认主源展示，AkShare 作为兜底，JQData/BaoStock 显示为预留或未配置。
- provider、frequency、adjust、任务状态、失败原因要清楚。
- 创建回测、发布快照、复制链接要有明确反馈。
- 长表格需要搜索、筛选、分页或固定滚动区域。
- 操作按钮要区分主动作和次动作，避免演示时找不到下一步。

### P4：跨端可靠性

目标：减少 demo 现场风险。

检查项：

- 桌面宽度、投屏宽度、移动宽度都不重叠、不溢出。
- 图表 canvas 非空，尺寸稳定，切换宽度后仍能正常渲染。
- 控制台没有应用级 error。
- API 失败时页面能显示原因。
- build 通过：`frontend-display` 和 `frontend-admin`。

## 设计约束

- 管理端应偏工作台风格：信息密度高、层级清楚、少装饰。
- 展示端应偏客户报告风格：可读、可信、能解释真实结果。
- 不做营销 landing page，第一屏直接服务实际使用。
- 不继续用一堆功能卡片简单拼管理台；页面分区应服务导航、任务流和模块职责。
- 不把卡片套卡片，不堆叠装饰容器。
- 不使用大面积单一色系或过度渐变背景。
- 文本不能和图表、按钮、表格重叠。
- 按钮文字要能在移动端容器内放下，必要时换行或缩短。
- 图标优先使用现有依赖：管理端已有 `@ant-design/icons`，不要随意新增图标库。
- 图表继续使用现有 `echarts` 和 `lightweight-charts`，不要无故换库。

## 建议验收

至少执行：

- 完成并提交本轮调研/设计草案，明确参考对象、采用点和不采用点。
- `cd frontend-display; npm.cmd run build`
- `cd frontend-admin; npm.cmd run build`
- 用浏览器打开当前本地展示端报告，检查桌面和移动宽度。
- 用浏览器打开管理端，检查数据、回测、快照、paper run 关键区域。
- 检查 console error。
- 如改动影响 API 类型、payload 字段或后端契约，必须先获得批准；批准后运行后端相关测试或至少说明未运行原因。

## 交付格式

完成后请说明：

- 本轮重构了哪些页面组织、模块布局和跳转路径。
- 本轮调研了哪些同类模块 UI，采用和放弃了哪些设计。
- 对后台演示效果有什么帮助。
- 对客户报告第一眼效果有什么帮助。
- 哪些旧快照或失败状态已兼容。
- 验证命令和浏览器检查结果。
- 仍保留哪些展示风险。

不要在最终答复中暴露真实密钥、token 或私有报告链接。
