# Customer Demo Talk Track

日期：2026-06-02

用途：客户演示时使用的讲解口径。本文档不是投顾合规文件，也不是商业合同。

## 1. 开场定位

这套系统当前定位是“内部量化研究与客户报告发布工作台”：

- 内部管理员负责接入行情、配置策略、运行回测、复核结果。
- 客户只看到已发布的只读报告。
- 报告快照不可变，分享链接可撤销。
- 系统当前不接券商，不做真实下单。

推荐说法：

> 这不是一个让客户在线调参或自动下单的交易终端，而是一个把研究、回测、复核和客户报告交付串起来的量化工作台。

## 2. 今天可以展示

- 数据源：Tushare Pro 主源，AkShare 免费兜底，JQData 和 BaoStock 保留 registry 位置。
- 行情导入：支持 Tushare 拉取和 CSV 导入，记录 provider、frequency、adjust、导入行数和请求参数。
- 数据质量：能提示样本过短、缺口、交易日历相关 warning。
- 回测：支持单标的和固定组合；包含资金、仓位、费用、滑点、基准、回撤和交易表。
- 指标：后端生成 MA、EMA、MACD、RSI、BOLL，并写入 snapshot。
- 策略解释：记录网格触发、MA 过滤、执行或拦截决策。
- 客户报告：展示核心指标、K 线证据图、交易记录、方法假设和风险披露。
- 模拟运行：管理端显示 paper run 状态流转、信号、模拟成交和失败原因。
- 运维治理：已有 Alembic baseline，默认 SQLite，后续可走 PostgreSQL/TimescaleDB。

## 3. 演示路径

推荐顺序：

1. 打开管理端，说明内部工作台和客户报告端分离。
2. 展示数据源和行情导入任务，强调 provider、复权和导入结果可追溯。
3. 选择参数集并运行回测。
4. 复核回测结果：收益、回撤、胜率、交易表、信号说明。
5. 发布 snapshot 并打开客户报告。
6. 展示 paper run：成功记录、失败记录、状态流转。
7. 回到风险披露，强调模拟、历史回测和非收益承诺。

## 4. 必须说清的边界

- 历史回测不代表未来收益。
- paper run 是模拟，不是实盘交易。
- 当前没有接入券商 API，也没有自动下单。
- 当前不是投顾合规系统。
- 当前默认数据库是 SQLite，生产化数据库路线已规划但未强制切换。
- JQData 暂未购买，不能说已接通。
- BaoStock 目前只预留 registry，不能说已实现。
- 分享链接 token 不应公开写入文档或截图说明。

## 5. 如何解释薄弱点

可以坦诚说明：

- 当前策略仍是规则型滚动做 T / 网格样例，不是策略市场。
- 指标解释已经后端化，但还没有做参数自动优化。
- 数据库已建立 migration baseline，PostgreSQL/TimescaleDB 是下一步生产化路线。
- 模拟运行已经可审计，但还没有接实时行情和实盘风控。
- 客户报告适合展示和沟通，不是实时交易指令。

推荐说法：

> 我们优先把“数据真实、回测可解释、报告可交付、边界清楚”做稳。接下来才适合扩展更多数据源、生产数据库和监控策略。

## 6. 外部审计吸收情况

来自 JZhu Trading 审计的可吸收点：

- TimescaleDB 适合长期分钟线，但当前不强制切换。
- 指标应该由后端统一生成，当前已完成第一批指标服务。
- backtest、monitor、paper run 应持久化历史，当前已强化回测和 paper run。
- 宽泛交易终端 UI 不适合当前 V1，当前仍保持“内部管理端 + 客户只读报告”。
- 云 AI 和策略市场不进入当前 V1，避免削弱审计和解释能力。

## 7. 演示结束收口

推荐收口：

> 当前版本已经能支撑一个可信的量化 demo：真实或固定行情导入、可解释回测、不可变客户报告、模拟运行审计和清晰风险披露。下一阶段的重点不是马上承诺实盘，而是接入更多已批准数据源、强化数据库部署和扩展监控策略。
